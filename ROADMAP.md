## Контекст проекта

### Зачем это нужно

Лаборатория LabDoctorM регулярно создаёт презентации: ADR, инциденты, отчёты, питчи. Сейчас это делается вручную или через разовые скрипты (как ПолИСКОП через fpdf2/python-pptx). Каждый раз — с нуля, без переиспользования, без стандартов.

**Проблемы текущего подхода:**
- Нет единой дизайн-системы — каждая презентация выглядит по-разному
- Нет переиспользования шаблонов — каждый раз пишем с нуля
- Нет визуального контроля — генерация вслепую, ошибки видны только после открытия
- Нет верификации — никто не проверяет читаемость, контрастность, структуру
- Нет документации — каждый агент изобретает свой подход

### Что мы хотим построить

Систему генерации презентаций, которая:
- Принимает данные (текст, заголовки, таблицы) → выдаёт PDF
- Имеет единую дизайн-систему (цвета, шрифты, отступы)
- Проверяет результат автоматически (OCR + структурный анализ)
- Итеративно улучшает при обнаружении проблем
- Масштабируется: один шаблон → много презентаций

### Аналоги (исследование рынка)

| Инструмент | Подход | Сильные стороны | Слабые стороны |
|---|---|---|---|
| **Gamma AI** | AI-first, card-based | 5 мин на презентацию, дизайнерское качество | Облачный, нет локального API |
| **Kimi Slides** | K2 Thinking + Nano Banana Pro | Два режима (контент/визуал), 4K визуал | Облачный, закрытый API |
| **Plus AI** | PPTX генерация | Редактируемый результат в PowerPoint | Только один формат |
| **Open Design** | 150+ дизайн-систем, MCP | Открытый код, 22 адаптера | Desktop-приложение, требует GUI |
| **python-pptx** | Программная генерация PPTX | Полный контроль | Сложный layout, блоки разъезжаются |
| **WeasyPrint** | HTML/CSS → PDF | Pixel-perfect, CSS Grid | Нет нативного PPTX |

**Наш выбор:** WeasyPrint + HTML/CSS + Jinja2 — потому что:
1. Работает на сервере (нет GUI-зависимостей)
2. CSS Grid/Flexbox решает проблему layout
3. Легко интегрируется с лабораторными пайплайнами
4. Поддерживает все возможности CSS (тени, градиенты, анимации)

## Фаза 1: Дизайнерский уровень

> **Цель:** Презентации выглядят чисто, профессионально, без явных ошибок.
> **Критерий принятия:** ЗавЛаб открывает PDF и говорит "это можно показывать".

### 1.1 Визуальный контроль (итеративность)

**Проблема:** Сейчас я генерирую PDF → загружаю на Диск → ЗавЛаб открывает → видит проблемы → говорит → я правлю. Это медленно и неэффективно.

**Решение:** Автоматизировать цикл проверки.

```
Генерация PDF → pdftoppm (скриншоты) → Mistral OCR (текст) → Анализ структуры → Отчёт о проблемах
```

**Конкретные шаги:**
1. Создать `verify.py` — скрипт который делает скриншоты каждого слайда
2. Отправлять скриншоты в Mistral OCR — проверять что весь текст читается
3. Анализировать структуру: есть ли заголовки, списки, таблицы
4. Формировать отчёт: какие слайды ок, какие имеют проблемы
5. Интегрировать в `generate.py` — автоматическая проверка после генерации

**Ожидаемый результат:** После запуска `python3 generate.py` автоматически получаем отчёт качестве.

### 1.2 Улучшение типографики

**Проблема:** Системные шрифты (DejaVu) выглядят дешёво. Нет брендинга.

**Решение:** Подключить шрифты референса.

**Конкретные шаги:**
1. Скачать Inter (Regular, Bold, Extrabold) и Oswald с Google Fonts
2. Разместить в `assets/fonts/`
3. Подключить через `@font-face` в CSS
4. Калибровка размеров для формата 16:9 (256mm × 144mm):
   - Заголовок слайда: 48-56px
   - Подзаголовок: 24-28px
   - Тело: 14-16px
   - Подписи: 10-12px
5. Межстрочный интервал: 1.4-1.6 для тела, 1.1-1.2 для заголовков
6. Letter-spacing: -0.02em для заголовков, 0 для тела

**Ожидаемый результат:** Презентация выглядит как с сайта polyscop.shtab-ai.ru.

### 1.3 Уменьшение пустого пространства

**Проблема:** Слайды выглядят пустыми — контент занимает ~50% площади.

**Причины:**
- Слишком большие отступы между элементами (margin-bottom: 56px)
- Карточки слишком маленькие по сравнению со слайдом
- Заголовок занимает слишком много места

**Решение:** Калибровка пропорций.

**Конкретные шаги:**
1. Заголовок слайда: не более 15% высоты слайда
2. Основной контент: 60-70% высоты слайда
3. Отступы между секциями: 16-24px (не 32-56px)
4. Карточки: минимум 200px высотой, заполненные текстом
5. Внутренние отступы карточек: 16-20px
6. Между карточками: 12-16px

**Ожидаемый результат:** Слайд заполнен на 70-80%, нет "дыр" между элементами.

### 1.4 Цветовые акценты и контрастность

**Проблема:** Не все тексты хорошо читаются. Нет визуальной иерархии через цвет.

**Решение:** Система цветовых ролей.

**Конкретные шаги:**
1. Первичный текст (#E6F1FF) — заголовки, ключевые фразы
2. Вторичный текст (rgba(230,241,255,0.6)) — описания, пояснения
3. Приглушённый текст (rgba(230,241,255,0.35)) — подписи, метаданные
4. Акцент (#00E5FF) — только ключевые слова, заголовки секций
5. Семантические цвета: зелёный (успех), жёлтый (внимание), красный (проблема)
6. Проверка контраста: все тексты должны иметь контраст ≥ 4.5:1 по WCAG AA

**Ожидаемый результат:** Чёткая визуальная иерархия, высокая читаемость.

## Фаза 2: Расширение системы шаблонов

> **Цель:** 17+ типов слайдов, данные отделены от шаблона.
> **Критерий принятия:** Можно сгенерировать презентацию из JSON-файла.

### 2.1 Библиотека шаблонов слайдов

**Проблема:** Сейчас захардкожен HTML — нельзя переиспользовать для других презентаций.

**Решение:** Выделить типы слайдов как отдельные шаблоны.

**Полный список шаблонов (31):**

| Шаблон | Описание | Варианты |
|---|---|---|
| hero | Титульный: большой заголовок + подзаголовок + CTA | split, minimal, bold |
| section | Разделитель секции | — |
| problem | Проблема: статистика + последствия | split |
| solution | Решение: преимущества | minimal |
| how | Пайплайн: шаги со стрелками | — |
| benefits | Выгоды: сетка карточек | list |
| economics | Экономика: цены + расчёт | — |
| competition | Сравнение: таблица | cards |
| market | Рынок: таблица с прогнозами | — |
| roadmap | Дорожная карта: временная шкала | timeline |
| team | Команда: карточки участников | grid |
| cta | Призыв к действию | bold |
| quote | Цитата: большой текст + автор | — |
| stats | Статистика: 3-4 большие цифры | big |
| timeline | Временная шкала | — |
| faq | Вопросы-ответы | accordion |
| contact | Контакты | — |
| toc | Содержание | — |
| government | Соответствие стратегии | — |

**Конкретные шаги:**
1. Создать `templates/slides/` директорию
2. Каждый тип слайда = отдельный Jinja2-шаблон
3. Варианты через `variant` поле: `hero-split.j2`, `hero-minimal.j2`, etc.
4. Базовый layout (`template.j2`) с `{% include %}` и автовыбором варианта

### 2.2 Данные через JSON

**Проблема:** Данные захардкожены в HTML. Чтобы изменить презентацию — нужно редактировать код.

**Решение:** Отделить данные от представления.

**Формат входных данных (JSON):**
```json
{
  "lang": "ru",
  "title": "ПОЛИСКОП",
  "subtitle": "AI-ассистент детекции полипов",
  "slides": [
    {
      "type": "hero",
      "variant": "bold",
      "title": "ПОЛИСКОП",
      "subtitle": "AI-ассистент...",
      "cards": [
        {"title": "Импортозамещение", "body": "..."},
        {"title": "Характеристики", "body": "..."}
      ]
    }
  ]
}
```

**Конкретные шаги:**
1. Схема данных (JSON Schema)
2. Парсер: JSON → контекст для Jinja2
3. CLI: `python3 generate.py --data presentation.json`
4. Валидация входных данных

### 2.3 Множественные презентации

**Проблема:** Один шаблон — одна презентация.

**Решение:** Один движок — много презентаций.

**Конкретные шаги:**
1. Директория `examples/` с примерами данных
2. Пакетная генерация: `python3 generate.py --batch examples/*.json`
3. Разные темы/брендбуки через CSS-переменные
4. Конфигурация через YAML-файл

## Фаза 3: Интеграция с лабораторией

> **Цель:** Проект встроен в рабочие процессы лаборатории.
> **Критерий принятия:** Любой агент может сгенерировать презентацию по ТЗ.

### 3.1 Скил presentation-maker

**Проблема:** Каждый агент не знает как делать презентации. Нет стандарта.

**Решение:** Создать кастомный скил.

**Содержимое скила:**
- Как создать презентацию: от ТЗ до PDF
- Какие шаблоны использовать для каких типов презентаций
- Как проверить качество (verify.py)
- Как загрузить на Яндекс Диск
- Стандарты: минимальный размер шрифта, контрастность, отступы

### 3.2 Автоматизация (GitHub Actions)

**Проблема:** Тесты запускаются только локально. При пуше — нет проверки.

**Решение:** CI/CD через GitHub Actions.

**Конкретные шаги:**
1. `.github/workflows/test.yml` — запуск тестов при каждом пуше
2. `.github/workflows/release.yml` — генерация PDF при релизе
3. `.github/workflows/preview.yml` — загрузка скриншотов как артефакты
4. Защита ветки main: тесты должны проходить перед мержем

### 3.3 Расширенная верификация

**Проблема:** Базовые тесты не ловят визуальные проблемы.

**Решение:** Многоуровневая проверка.

**Уровни верификации:**
1. **Структурная:** HTML валиден, все секции на месте
2. **Текстовая:** OCR распознаёт весь текст, нет пустых слайдов
3. **Визуальная:** Контрастность ≥ 4.5:1, нет наложения элементов
4. **Семантическая:** Заголовки есть, списки корректны, таблицы читаемы

## Фаза 4: Продвинутые функции

> **Цель:** Поддержка множественных форматов и AI-улучшений.
> **Критерий принятия:** Можно экспортировать в PPTX, HTML, PNG.

### 4.1 Экспорт в разные форматы

**Проблема:** Только PDF. Иногда нужен PPTX для редактирования или HTML для браузера.

**Решение:** Мультиформатный экспорт.

**Конкретные шаги:**
1. PPTX: python-pptx (с фиксированной сеткой, не абсолютные координаты)
2. HTML: тот же шаблон, но с навигацией и анимациями
3. PNG: pdftoppm с настраиваемым DPI
4. Единый CLI: `python3 generate.py --format pdf|pptx|html|png`

### 4.2 Интерактивные презентации (HTML)

**Проблема:** PDF статичен. Нет анимаций, навигации, интерактивности.

**Решение:** HTML-презентация как отдельный формат.

**Конкретные шаги:**
1. Клавиатурная навигация (стрелки, пробел)
2. Анимации появления (CSS transitions)
3. Прогресс-бар внизу
4. Содержание (оглавление с ссылками)
5. Полноэкранный режим (Fullscreen API)

### 4.3 AI-улучшения

**Проблема:** Компоновка слайда ручная. LLM может помочь.

**Решение:** Интеграция с LLM для генерации контента.

**Конкретные шаги:**
1. Генерация текста слайда по теме (через api-hub)
2. Автоматический подбор шаблона по типу контента
3. Умная композиция: LLM предлагает layout по тексту
4. A/B тестирование: два варианта → выбор лучшего

### 4.4 Совместная работа

**Проблема:** Презентации создаются в одиночку. Нет ревью, комментариев.

**Решение:** Базовая система совместной работы.

**Конкретные шаги:**
1. Версионирование презентаций в git
2. Pull Request для ревью презентаций
3. Комментарии к слайдам (через GitHub Issues)
4. Экспорт на Яндекс Диск для шаринга

## Фаза 5: Платформа

> **Цель:** Веб-интерфейс для создания презентаций без кода.
> **Критерий принятия:** Пользователь создаёт презентацию через браузер за 5 минут.

### 5.1 Web-интерфейс

**Проблема:** Сейчас нужно писать код и JSON. Неудобно для не-разработчиков.

**Решение:** Простой веб-редактор.

**MVP функциональность:**
- Список слайдов слева
- Редактор контента справа (WYSIWYG или markdown)
- Предпросмотр в реальном времени
- Выбор шаблона из библиотеки
- Экспорт в PDF/PPTX одной кнопкой

**Конкретные шаги:**
1. React + Vite + Tailwind (как в проекте polyscope)
2. Компоненты: SlideEditor, Preview, TemplateSelector, ExportPanel
3. API для генерации PDF на сервере (FastAPI)
4. Деплой на GitHub Pages или отдельный сервер

### 5.2 REST API

**Проблема:** Нет программного доступа к генератору.

**Решение:** API для интеграций.

**Эндпоинты:**
```
POST /api/v1/generate — генерация презентации
GET  /api/v1/templates — список шаблонов
GET  /api/v1/status/{id} — статус генерации
GET  /api/v1/download/{id} — скачивание PDF
```

**Конкретные шаги:**
1. FastAPI приложение
2. Очередь задач (Celery или asyncio)
3. Кэширование результатов
4. Rate limiting

### 5.3 Аналитика

**Проблема:** Не знаем какие шаблоны популярны, сколько времени занимает генерация.

**Решение:** Базовая аналитика.

**Метрики:**
- Количество сгенерированных презентаций в день
- Популярные шаблоны
- Среднее время генерации
- Ошибки и сбои
